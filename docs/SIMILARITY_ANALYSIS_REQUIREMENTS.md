# 유사도 분석 기반 기밀데이터 탐지 설명 자료

## 1. 개요

본 기능은 데이터 분류 시스템에서 기밀데이터로 식별한 사내 문서, 텍스트, 임베딩 벡터 정보를 모니터링 솔루션과 연계하여, 네트워크 트래픽 및 첨부파일 내 기밀데이터 유출 가능성을 탐지하는 체계이다.

모니터링 솔루션은 제공받은 기밀데이터 임베딩 벡터와 실시간 모니터링 대상 데이터를 비교하여, 설정된 유사도 임계치 이상인 경우 기밀데이터와 유사한 정보가 포함된 것으로 판단하고 탐지 이벤트로 분류한다.

### 문서 표기 기준

본 문서는 요구사항과 현재 `xcn-similarity` 구현 상태를 함께 설명한다. 외부 설명 시 혼선을 줄이기 위해 각 항목은 다음 기준으로 구분한다.

| 표기 | 의미 |

| TODO | 요구사항으로 정의되어 있으나 추가 구현 또는 외부 시스템 연계가 필요한 기능 |

## 2. 기본 동작 구조

목표 구조에서는 데이터 분류 시스템이 사내 문서 및 텍스트 데이터를 분석하여 기밀데이터 여부를 분류한다.

기밀데이터로 분류된 데이터는 다음 정보를 모니터링 솔루션에 제공한다.

- 기밀데이터 식별 정보
- 기밀데이터 분류 라벨
- 청크 단위 설명 라벨
- 임베딩 벡터
- 벡터 DB 등록에 필요한 메타데이터

모니터링 솔루션은 제공받은 임베딩 벡터를 기준으로 트래픽 및 첨부파일 데이터를 분석한다.

분석 대상 데이터는 청킹 및 임베딩 과정을 거쳐 벡터화되며, 벡터 DB에 등록된 기밀데이터 벡터와 유사도 비교를 수행한다.

유사도가 설정 임계치 이상이면 기밀데이터 탐지로 판단한다.

현재 `xcn-similarity`는 외부 데이터 분류 시스템에서 임베딩 벡터를 직접 수신하는 전용 연계 API는 아직 구현되어 있지 않다. 대신 등록문서를 직접 업로드하거나 텍스트를 입력하면 시스템 내부에서 텍스트 추출, 청킹, 임베딩, 벡터 DB 등록을 수행하고, EMS 로그 본문 및 첨부파일과 유사도 비교를 수행한다.

### 3.1 기밀데이터 벡터 연계

데이터 분류 시스템에서 기밀데이터로 분류한 사내 문서, 텍스트, 임베딩 벡터 DB를 모니터링 솔루션에 제공해야 한다.

모니터링 솔루션은 제공받은 임베딩 벡터를 기준으로 유사도 분석을 수행한다.

설정된 유사도 임계치 이상으로 매칭되는 경우, 해당 모니터링 대상 데이터는 기밀데이터 포함 가능성이 있는 것으로 탐지한다.

현재 상태: TODO

현재 구현은 외부 데이터 분류 시스템이 생성한 임베딩 벡터를 직접 수신하는 방식이 아니라, 등록문서 또는 입력 텍스트를 `xcn-similarity` 내부 임베딩 모델로 벡터화하여 Milvus에 등록하는 방식이다. 외부 분류 시스템 연계를 위해서는 벡터, 라벨, 청크 설명 정보를 수신하는 API 또는 배치 동기화 기능이 추가되어야 한다.

### 3.2 자체 임베딩 및 청킹 기능

모니터링 솔루션은 데이터 분류 시스템에서 제공받지 않은 기밀데이터 또는 신규 데이터도 처리할 수 있어야 한다.

이를 위해 별도의 기능을 제공해야 한다.

- 텍스트 청킹
- 첨부파일 텍스트 추출
- 임베딩 생성
- 벡터 DB 등록
- 라벨 및 메타데이터 관리

즉, 외부에서 제공된 벡터만 사용하는 구조가 아니라, 모니터링 솔루션 자체에서도 기밀데이터 후보를 벡터화하고 관리할 수 있어야 한다.

현재 상태: 현재 제공

현재 구현은 등록문서 업로드, 텍스트 입력, 파일 기반 검색, EMS 로그 본문 및 첨부파일 색인 과정에서 자체 텍스트 추출, 청킹, 임베딩을 수행한다.

### 3.3 유사도 임계치 조정

모니터링 솔루션은 유사도 탐지 임계치를 운영자가 조정할 수 있어야 한다.

예를 들어 다음과 같은 기준을 설정할 수 있다.

| 구분 | 기준 예시 | 설명 |
| --- | --- | --- |
| 고위험 탐지 | 유사도 0.85 이상 | 기밀데이터 포함 가능성이 높은 구간 |
| 일반 탐지 | 유사도 0.72 이상 | 탐지 이벤트로 분류할 수 있는 구간 |
| Grey Zone | 유사도 0.60 이상 0.72 미만 | 운영자 수동 검토가 필요한 구간 |

임계치는 운영 정책, 탐지 민감도, 오탐률, 업무 특성에 따라 조정 가능해야 한다.

현재 상태: 현재 제공

현재 운영 화면에서는 검색 시 `Min Score`와 `Top K`를 조정할 수 있다. 최근 고위험 매칭 기준은 `SIM_RECENT_MATCH_MIN_SCORE` 환경 변수로 조정할 수 있다.

### 3.4 Grey Zone 수동 분류

유사도 기준상 명확한 탐지 또는 정상으로 판단하기 어려운 영역은 Grey Zone으로 분류한다.

Grey Zone에 해당하는 데이터는 운영자가 수동으로 검토하고 다음과 같이 분류할 수 있어야 한다.

- 기밀데이터 탐지 확정
- 정상 데이터
- 예외 처리
- 추가 검토 필요
- 라벨 보정

이를 통해 유사도 기반 탐지의 오탐과 미탐을 줄이고, 운영 과정에서 탐지 품질을 개선할 수 있다.

현재 상태: 대체 제공/TODO

현재 구현은 유사도 점수, 등록문서/로깅데이터 비교, 공통 핵심어구, 매칭 근거 팝업을 통해 운영자가 Grey Zone 후보를 검토할 수 있도록 지원한다. 다만 검토 결과를 `탐지 확정`, `정상`, `예외`, `추가 검토` 등으로 저장하고 이력을 관리하는 전용 수동 분류 워크플로는 추가 구현이 필요하다.

### 3.5 원본데이터 미저장 원칙

임베딩 벡터 DB에는 원본데이터를 저장해서는 안 된다.

기밀데이터 유출 위험을 줄이기 위해, 원본 문서 및 원문 텍스트는 임베딩 생성 후 삭제해야 한다.

벡터 DB에는 다음 정보만 저장하는 것을 원칙으로 한다.

- 임베딩 벡터
- 문서 또는 청크 식별자
- 분류 라벨
- 설명 라벨
- 보안 등급
- 등록 일시
- 관리 메타데이터

원문 데이터는 저장하지 않으며, 탐지 사유 소명을 위해 필요한 최소한의 설명 정보만 라벨 형태로 관리한다.

현재 상태: 대체 제공/TODO

현재 구현은 Milvus 벡터 DB와 MongoDB 카탈로그를 분리하여 운영한다. Milvus에는 벡터와 검색용 메타데이터 중심으로 저장한다. 다만 운영 화면에서 문서/로그 청크 조회와 본문 비교를 제공하기 위해 일부 텍스트가 카탈로그 또는 검색 결과에 보관될 수 있으므로, 요구사항의 `임베딩 후 원본 자동 삭제`를 완전히 만족하려면 원문 미보관 모드, 청크 텍스트 삭제 정책, 라벨 중심 소명 구조를 추가로 고도화해야 한다.

### 3.6 탐지 사유 설명 라벨

원본데이터를 저장하지 않더라도 탐지 사유를 설명할 수 있어야 한다.

이를 위해 데이터 분류 시스템은 청크 단위 설명 라벨을 제공해야 한다.

예시는 다음과 같다.

```text
인사팀에서 연봉정보로 선정한 기밀데이터에서 생성된 임베딩 벡터와 유사도 0.72 이상으로 매칭되어 탐지
```

모니터링 솔루션은 외부에서 제공받은 벡터뿐 아니라, 자체 임베딩한 벡터에도 동일하게 설명 라벨을 부여할 수 있어야 한다.

이를 통해 탐지 이벤트 발생 시 운영자가 다음 내용을 확인할 수 있다.

- 어떤 기밀데이터 기준과 유사한지
- 어떤 보안 라벨에 해당하는지
- 유사도 점수는 얼마인지
- 탐지 사유는 무엇인지
- 수동 검토가 필요한지 여부

현재 상태: 대체 제공/TODO

현재 구현은 문서 제목, 보안 등급, 문서/로그 식별자, 유사도 점수, 공통 핵심어구, 등록문서/로깅데이터 비교를 제공하여 탐지 근거 확인을 지원한다. 다만 외부 데이터 분류 시스템에서 제공하는 청크 단위 설명 라벨을 표준 필드로 수신하고, 원문 없이 라벨만으로 소명하는 기능은 TODO이다.

### 3.7 벡터 DB 삭제 기능

벡터 DB는 등록된 기밀데이터 벡터에 대한 삭제 기능을 지원해야 한다.

삭제 기능은 다음 상황에 필요하다.

- 기밀데이터 분류 해제
- 잘못 등록된 데이터 정정
- 보안 정책 변경
- 보존 기간 만료
- 라벨 오류 수정
- 사용자 요청 또는 운영자 조치

삭제 시에는 벡터뿐 아니라 관련 메타데이터 및 라벨 정보도 함께 정리되어야 한다.

현재 상태: 현재 제공

현재 구현은 등록문서 삭제 기능을 통해 문서 카탈로그와 관련 벡터를 정리하는 기능을 제공한다. 외부 분류 시스템과 연동되는 경우에는 외부 분류 해제 이벤트와 동기화하는 삭제 API 또는 배치 삭제 절차가 추가로 필요하다.

### 3.8 모니터링 대상

모니터링 대상은 네트워크 또는 업무 시스템을 통해 전송되는 데이터이다.

주요 대상은 다음과 같다.

- POST 등 HTTP 요청으로 전송되는 본문 데이터
- API 요청 페이로드
- 웹 업로드 데이터
- 메일 또는 업무 시스템 첨부파일
- 문서 파일
- 텍스트 파일
- 압축 해제 또는 문서 변환 후 추출 가능한 텍스트

첨부파일은 파일 유형별로 텍스트를 추출한 뒤 청킹 및 임베딩을 수행하여 유사도 분석 대상에 포함한다.

현재 상태: 대체 제공/TODO

현재 구현은 EMS 원천 데이터의 메시지 본문과 첨부파일 또는 첨부 추출 텍스트를 색인한다. POST 등 실시간 네트워크 트래픽을 모니터링 솔루션에서 직접 수집하여 즉시 유사도 분석하는 연계는 별도 구현 대상이다.

## 4. 처리 흐름

### 4.1 목표 처리 흐름

1. 데이터 분류 시스템이 사내 문서 및 텍스트를 분석한다.
2. 기밀데이터로 분류된 항목을 청크 단위로 분할한다.
3. 각 청크에 대해 임베딩 벡터를 생성한다.
4. 원본데이터는 임베딩 후 삭제한다.
5. 벡터, 라벨, 설명 정보, 메타데이터를 모니터링 솔루션에 제공한다.
6. 모니터링 솔루션은 제공받은 벡터를 벡터 DB에 등록한다.
7. 모니터링 대상 트래픽과 첨부파일을 수집한다.
8. 수집 데이터에서 텍스트를 추출하고 청킹한다.
9. 각 청크를 임베딩한다.
10. 벡터 DB에 등록된 기밀데이터 벡터와 유사도를 비교한다.
11. 임계치 이상이면 탐지 이벤트로 분류한다.
12. Grey Zone 데이터는 운영자가 수동 분류한다.
13. 탐지 결과는 유사도 점수, 라벨, 설명 정보와 함께 제공한다.

목표 처리 흐름 중 1번부터 5번까지의 외부 데이터 분류 시스템 연계, 12번의 수동 분류 결과 저장 워크플로, 13번의 청크 단위 설명 라벨 기반 소명 기능은 TODO 항목이다.

### 4.2 현재 구현 처리 흐름

현재 `xcn-similarity`는 다음 흐름을 제공한다.

1. 운영자가 등록문서를 업로드하거나 텍스트를 입력한다.
2. 시스템이 문서 또는 텍스트에서 본문을 추출한다.
3. 추출 텍스트를 청크 단위로 분할한다.
4. `upskyy_bge_m3_korean` 모델로 청크 임베딩을 생성한다.
5. 생성된 벡터를 Milvus에 등록한다.
6. 문서 식별자, 보안 등급, 청크 수, 운영 메타데이터를 MongoDB 카탈로그에 저장한다.
7. `xcn-similarity-indexer`가 EMS MongoDB와 MinIO에서 로그 본문 및 첨부 데이터를 조회한다.
8. EMS 본문 및 첨부 텍스트를 청킹/임베딩하여 로그 벡터로 등록한다.
9. 유사도 검색 화면에서 문서 기준 로그 추적 또는 로그 기준 등록문서 매칭을 수행한다.
10. 운영자가 `Min Score`, `Top K`, 대상 필터를 조정하여 후보군을 조회한다.
11. 선택 결과의 유사도 점수, 공통 핵심어구, 등록문서/로깅데이터 비교 화면을 통해 탐지 근거를 확인한다.
12. 필요 시 매칭 근거 팝업으로 상세 내용을 확인한다.

현재 구현은 유사도 분석과 근거 확인을 제공하지만, Grey Zone 수동 분류 결과를 저장하는 별도 상태 관리 기능은 제공하지 않는다.

```json
  "embedding_dim": 1024
}
```

해당 구성은 한국어 문서 및 로그 텍스트의 의미 유사도 비교를 위해 사용된다. 문서와 로그는 동일한 임베딩 모델로 벡터화되어 동일 벡터 공간에서 비교된다.

### 5.9 원본 보관 정책과 현재 구현 기준

요구사항상 임베딩 벡터 DB에는 원본데이터를 저장하지 않는 것이 원칙이다. 현재 구현에서는 벡터 DB와 카탈로그를 분리하여 운영하며, Milvus에는 벡터와 검색용 메타데이터 중심으로 저장한다.

| 저장 위치 | 저장 내용 |

## 테스트 개요

- 테스트 일시: 2026-07-01
- 대상 서버: 52번 서버 `10.100.40.52`
- 대상 API: `http://127.0.0.1:8010`
- 임베딩 백엔드: `hf_transformer`
- 임베딩 모델: `upskyy_bge_m3_korean`
- 임베딩 차원: `1024`
- 벡터 DB: Milvus
- 측정 스크립트: `tools/perf_similarity_api.py`

## 테스트 방식

1. 현재 적재 데이터 기준 베이스라인 측정
2. 합성 문서 100건 등록 후 동일 조건 측정
3. 합성 문서 1,000건 등록 후 동일 조건 측정
4. 합성 문서 10,000건 등록 후 동일 조건 측정
5. 테스트 문서는 측정 후 삭제

합성 문서는 `/similarity/documents` API로 등록했고, `metadata.source=perf_test`, `metadata.file_retained=false`로 생성했다. 테스트 후 등록한 문서는 모두 삭제했다.

### 등록문서 기준 로그 검색 최적화 후

적용 내용:

- 로그 검색 기본 기간을 최근 30일로 제한하고, 해당 기간의 월별 Milvus partition을 지정한다.
- 등록문서 기준 로그 검색에 사용하는 문서 청크 수를 옵션(`SIM_SEARCH_LOGS_MAX_DOCUMENT_CHUNKS`, 기본 8)으로 제한한다.
- 여러 문서 청크 벡터를 Milvus batch search로 조회한다.
- batch search 실패 시 `SIM_SEARCH_LOGS_PARALLELISM` 기준 병렬 검색으로 fallback한다.
- 동일 요청은 `SIM_SEARCH_LOGS_CACHE_TTL_SEC` 동안 MongoDB match cache를 사용한다.

기본 옵션:

|설정|기본값|설명|
|---|---:|---|
|`SIM_SEARCH_LOGS_DEFAULT_DAYS`|30|요청에 `metadata_filter.ctime`이 없을 때 최근 N일만 검색|
|`SIM_SEARCH_LOGS_MAX_DOCUMENT_CHUNKS`|8|등록문서 기준 로그 검색에 사용할 최대 문서 청크 수|
|`SIM_SEARCH_LOGS_PARALLELISM`|4|batch search 실패 시 fallback 병렬 검색 수|
|`SIM_SEARCH_LOGS_CACHE_ENABLED`|true|동일 조건 검색 결과 캐시 사용|
|`SIM_SEARCH_LOGS_CACHE_TTL_SEC`|300|검색 결과 캐시 TTL|
|`SIM_SEARCH_LOGS_INCLUDE_DEFAULT_PARTITION`|true|월 partition과 함께 `_default` partition 포함|

## 해석

- 등록문서 유사도 검색은 문서 청크 320개에서 2,320개로 늘릴 때는 p95가 약 75ms에서 72ms 수준으로 유지됐다.
- 합성 문서 10,000건, 문서 청크 20,320개 기준에서는 등록문서 유사도 검색 p95가 약 92ms로 증가했다. 그래도 100ms 이내 범위로 유지됐다.
- 현재 규모에서는 문서 검색 병목이 Milvus row count보다는 API 처리, query 임베딩, 네트워크/스케줄링 영향에 더 가깝다.
- 문서 목록 조회는 문서 수가 27건에서 1,027건으로 늘면 p95가 7.5ms에서 38.5ms로, 10,027건에서는 80.4ms로 증가했다. 검색 API보다 MongoDB catalog 목록 조회/정렬/페이지 처리 영향이 더 민감하게 나타났다.
- 로그 텍스트 유사도 검색은 약 98.5만 로그 청크가 적재된 상태에서도 p95 44ms 수준이었다. Milvus 로그 컬렉션 자체 검색은 충분히 빠르다.
- 등록문서 기준 로그 검색(`/similarity/search/logs`)은 문서 청크를 기준으로 로그 컬렉션을 반복 검색하므로 단순 텍스트 로그 검색보다 무겁다. 최근 30일 월 partition, 문서 청크 제한, batch search, 캐시를 적용하면 동시성 3 기준 캐시 회피 p95 약 98ms, 캐시 적용 p95 약 8ms 수준으로 개선된다.

## 결론

현재 52번 서버 기준으로는 등록문서 10,000건, 문서 청크 20,320개 수준에서도 등록문서 유사도 검색 p95가 약 92ms로 유지됐다. 로그 청크 98.8만 개 대상 텍스트 검색도 p95 약 45ms로 안정적이다. 등록문서 기준 로그 검색은 기본 기간/partition/문서 청크 제한/batch search/cache를 적용한 운영 기본값에서 p95 100ms 안팎으로 동작한다.
